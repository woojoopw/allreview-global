## 개발 워크플로우 원칙

**핵심: 계획을 내가 승인하기 전까지 절대 코드를 쓰지 마라.**

1. **리서치** → research.md 작성 (구현 금지)
2. **계획** → plan.md 작성 (구현 금지)
3. **내가 검토 + 메모** → 반영 후 업데이트 (구현 금지)
4. **승인 후 구현** → plan.md 항목 체크하며 끝까지 실행

잘못된 방향이면 고치지 말고 git reset 후 재시작.

## 🔒 보안 필수 원칙 (무조건 지켜라)

> **AI가 생성한 코드의 45%에 보안 취약점이 존재한다.**
> 러버블, 앤리치리드, 레플릿 — 보안 검토 없이 출시했다가 개인정보 유출, 해킹, DB 삭제 사고가 실제로 발생했다.

### 절대 금지
- API 키, 비밀번호, 시크릿을 코드에 직접 하드코딩하지 마라
- `.env` 파일을 절대 git에 커밋하지 마라 (`.gitignore`에 반드시 포함)
- 에러 메시지에 시스템 내부 정보(스택 트레이스, DB 구조 등)를 노출하지 마라

### SQL Injection 방지
- 사용자 입력을 SQL 쿼리에 직접 붙이지 마라
- 반드시 파라미터화된 쿼리(Prepared Statement) 사용
- ORM 사용 시 raw query 최소화

### XSS (크로스 사이트 스크립팅) 방지
- 사용자 입력을 HTML에 그대로 렌더링하지 마라
- dangerouslySetInnerHTML 사용 금지
- 외부 데이터는 항상 escape 처리

### 인증 / 권한
- 모든 API 엔드포인트에서 인증 확인 필수
- 다른 유저의 데이터에 접근 못하도록 권한 체크
- 세션/토큰 만료 처리 필수

### 암호화
- 비밀번호는 반드시 bcrypt 등 안전한 해시 함수 사용
- 오래된/취약한 암호화 알고리즘 사용 금지

### 환경 분리
- 테스트 DB와 운영 DB 반드시 분리
- AI가 자동으로 DB를 정리/삭제하도록 두지 마라
- 개발 환경에서 운영 데이터 절대 사용 금지

### 라이브러리 관리
- 새 라이브러리 추가 시: `npm audit` 으로 취약점 확인
- 기능 추가할 때마다 `npm audit` 실행 습관화
- 유명 라이브러리인 척 악성 코드가 심어진 경우 있음 → 출처 확인

### 보안 로그
- 로그에 비밀번호, API 키, 개인정보 절대 기록하지 마라
- 에러 로그를 클라이언트에 그대로 노출하지 마라

### 코드 작성 후 보안 자가 검토 (매번 실행)
새 기능 구현 후 반드시 아래 명령어로 검토:
```
npm audit
```
그리고 Claude에게 이 코드에 SQL Injection, XSS, 인증 취약점이 있는지 검토해줘 라고 물어봐라.

## 기술 스택
- Next.js 14 (App Router) + TypeScript + Tailwind CSS
- Google Places API (전세계 장소 + 리뷰)
- next-intl (12개국 다국어)
- Vercel (배포)

## 환경변수 (Vercel에 이미 설정됨 — 코드에 직접 쓰지 마라)
- `NEXT_PUBLIC_GOOGLE_PLACES_API_KEY` — Google Places API 키
